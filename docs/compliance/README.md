## Why We Publish These Reports

Yashigani is a security product. Our customers trust us to protect their AI
infrastructure, and that trust must be earned — not assumed. We believe the best
way to demonstrate our security posture is to assess ourselves against the same
frameworks our customers are audited against, and to publish the results openly.

Every release of Yashigani undergoes automated compliance assessment against 11
internationally recognised frameworks totalling 1,275 controls. We publish the
results — including our failures — because we believe transparency is a
stronger signal than marketing claims.

If a control fails, we document it. If we cannot verify a control through
automated analysis alone, we mark it NEEDS REVIEW rather than claim a false
pass. If a control genuinely does not apply to our architecture (e.g. physical
access controls for a software product), we mark it N/A with a justification.

## How to Read These Reports

### Report Status

All reports carry a status:

- **DRAFT** — Machine-generated by YCS, pending manual review by our GRC team.
  The automated checks are real and the findings are accurate, but a human
  auditor has not yet verified edge cases or contextual nuances.
- **REVIEWED** — Manually reviewed by our Senior GRC Auditor (Lu). Every PASS
  has been verified against actual code with file paths and line numbers cited.
  Every FAIL has been confirmed as a genuine gap with remediation guidance.
- **PUBLISHED** — Final version, suitable for sharing with auditors and
  customers. Reviewed, corrected, and signed off by the compliance team.

### Control Verdicts

Each control receives one of four verdicts:

| Verdict | Meaning | What We Did |
|---------|---------|-------------|
| **PASS** | Control requirement is met | Verified against actual source code. Evidence cites specific file paths and, where applicable, line numbers. |
| **FAIL** | Control requirement is not met | Documented the gap and what would be needed to satisfy it. Prioritised by risk. |
| **NEEDS REVIEW** | Partial evidence found | Automated checks found relevant code, but the control requires human judgement to confirm full compliance (e.g. procedural controls, organisational policies). |
| **N/A** | Control does not apply | Justified why. For example, Yashigani is software delivered as containers — physical security controls for data centres do not apply to our product, though they apply to our customers' deployments. |

### What "Pass Rate" Means

The pass rate in the summary table is calculated as:

    Pass Rate = PASS / (PASS + FAIL) × 100

N/A controls are excluded from the denominator. A 92% pass rate means 92% of
*applicable* controls are satisfied, not that we ignored 8% of the framework.

### What These Reports Are NOT

- **Not a penetration test.** These reports verify the presence and
  configuration of security controls. They do not test whether those controls
  can be bypassed by a skilled attacker. We recommend regular penetration
  testing by an independent firm.
- **Not a certification.** Only accredited certification bodies can issue ISO
  27001 certificates, SOC 2 reports, or PCI DSS attestations. These reports
  provide evidence that can accelerate your certification process.
- **Not a guarantee.** Software security is a continuous process. These reports
  reflect the state of the codebase at a point in time. We re-run them before
  every release.

## Assessment Methodology

### Automated Checks (YCS)

The Yashigani Compliance Suite maps each framework control to one or more
verifiable checks against the codebase:

| Check Type | Description |
|---|---|
| `file_contains` | Read a specific file and verify a regex pattern is present |
| `any_file_contains` | Search a directory tree for a regex pattern |
| `file_exists` | Check that a file or glob pattern matches |
| `none_file_contains` | Verify a pattern is absent (negative check — e.g. no hardcoded secrets) |
| `all_of` | All sub-checks must pass (AND logic) |
| `any_of` | At least one sub-check must pass (OR logic) |
| `always_pass` | Control is not applicable — marked N/A with documented justification |
| `always_fail` | Known gap — marked FAIL with explanation and remediation plan |

Checks are deterministic: the same codebase always produces the same results.
There is no AI interpretation or probabilistic scoring.

### Manual Review (Lu)

For REVIEWED and PUBLISHED reports, our Senior GRC Auditor manually verifies:

1. Every PASS verdict has specific code evidence (not just pattern matching)
2. Every FAIL is a genuine gap (not a false positive from the scanner)
3. Every N/A has a valid architectural justification
4. NEEDS REVIEW items are resolved to PASS, FAIL, or N/A with reasoning
5. Findings are prioritised by actual risk to the deployment, not alphabetically

### Limitations

Automated compliance scanning has inherent limitations:

- **Code presence does not equal code correctness.** A password hashing
  function exists (PASS) but the scanner cannot verify it uses adequate
  cost parameters without reading the implementation.
- **Organisational controls cannot be automated.** Policies like "security
  awareness training" or "background checks" require organisational evidence
  that lives outside the codebase.
- **Runtime behaviour differs from static analysis.** Configuration files
  define intent; runtime testing confirms behaviour. These reports verify
  intent.
- **Self-assessment bias.** We assess our own product. We mitigate this by
  using deterministic automated checks (not subjective judgement), publishing
  failures alongside passes, and recommending independent verification.

## Pre-Release Process

Before every Yashigani release:

1. YCS runs all 1,275 controls against the release candidate
2. Any new FAILs are triaged — critical/high must be fixed before release
3. Reports are regenerated with updated findings
4. Senior GRC Auditor reviews changes from previous release
5. Reports are committed to the repository as part of the release

This process is documented in our Standard Operating Procedures and is
mandatory — no release ships without a compliance pass.

## For Auditors

If you are auditing an organisation that deploys Yashigani, these reports
provide a starting point for your assessment of the product's security
controls. We recommend:

1. **Verify our claims.** Every PASS cites a file path. Read the code yourself.
2. **Check the version.** These reports are version-specific. Confirm your
   customer's deployed version matches the report version.
3. **Assess the deployment.** Our reports cover the product. Your customer's
   deployment environment (network, host OS, access controls, backup
   procedures) is outside our scope.
4. **Contact us.** We welcome questions from auditors. Reach out at
   compliance@agnosticsec.com.

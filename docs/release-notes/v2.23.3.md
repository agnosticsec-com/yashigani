## Security

### ASVS V2.1.7 — HIBP k-Anonymity breach check

**OWASP ASVS V2.1.7:** "Verify that passwords submitted during account registration, login, and password change are checked against a set of breached passwords."

**Implementation:**

1. On every user-chosen password change, the proposed password is SHA-1 hashed.
2. The first 5 hex characters of the hash are sent to `api.pwnedpasswords.com/range/{prefix}` (k-Anonymity: only the prefix, never the full hash or password, is transmitted).
3. The API response contains all hash suffixes matching that prefix. If the full hash suffix is found, the password has appeared in a known breach.
4. A matching password is rejected with **HTTP 422 Unprocessable Entity** and the message: _"This password has appeared in known data breaches; choose another."_
5. Non-matching passwords proceed to Argon2id hashing and storage as normal.

**Scope:** user-chosen passwords only. System-generated temporary passwords (created at account bootstrap, admin-forced reset, or full account reset) are exempt — they are randomised with a 36-char secure alphabet and are not user-chosen.

**Failure mode (fail-open):** if the HIBP API is unreachable (network timeout, 5xx, DNS failure), the password change is allowed to proceed. A `WARNING` log entry is emitted and the `yashigani_hibp_check_api_unavailable_total` Prometheus counter is incremented. This preserves operator access during API outages.

**Privacy:** the full password and full SHA-1 hash are never transmitted. Only the first 5 characters of the SHA-1 hex digest leave the system. SHA-1 is used because it is mandated by the HIBP k-Anonymity wire protocol (NIST SP 800-63B §5.1.1.2); it is not used as a storage primitive (passwords are stored with Argon2id).

**Operator controls:**

| Environment variable | Default | Description |
|---|---|---|
| `YASHIGANI_HIBP_CHECK_ENABLED` | `true` | Set to `false` to disable HIBP checks. For air-gapped deployments. Disabling is a risk-accept against ASVS V2.1.7 — document in your risk register. |
| `YASHIGANI_HIBP_API_URL` | `https://api.pwnedpasswords.com/range/` | Override the API base URL to point at an internal HIBP mirror. Include trailing slash. |
| `YASHIGANI_HIBP_API_KEY` | _(none)_ | Optional API key for commercial-tier rate-limit lift or mirror auth. Prefer admin-panel configuration (see below). |

**Network requirement:** `backoffice` container requires outbound HTTPS to `api.pwnedpasswords.com:443` on the `edge` network (default topology). Use `YASHIGANI_HIBP_API_URL` to redirect to an internal mirror for air-gapped deployments.

**Observability:** the `yashigani_hibp_check_api_unavailable_total` Prometheus counter tracks failed HIBP API calls (each failure = one password change where the breach check was skipped due to API unreachability). Alert on sustained non-zero rate to detect loss of outbound edge connectivity.

### Admin-panel HIBP API key (v2.23.3 extension to PR #59)

Operators who need the HIBP commercial-tier rate-limit lift, or who run a self-hosted HIBP mirror requiring authentication, can now set and rotate the HIBP API key from the admin panel — **no container restart required**.

**Admin UI location:** Settings → Authentication > HIBP API Key

**Resolution priority (highest to lowest):**
1. Admin-panel key — encrypted at rest with the deployment AES key (`YASHIGANI_DB_AES_KEY`); takes effect immediately.
2. Env var `YASHIGANI_HIBP_API_KEY` — fallback for automation.
3. None — anonymous k-Anonymity request (free tier; works for most deployments).

**API endpoints:**
- `GET /api/v1/admin/auth/hibp/status` — returns `{configured, source, masked_value, updated_at, updated_by}`. Admin session required. Full key never returned.
- `PUT /api/v1/admin/auth/hibp/key` — set key. Body: `{api_key: "..."}`. Step-up TOTP required (ASVS V6.8.4).
- `DELETE /api/v1/admin/auth/hibp/key` — clear key. Step-up TOTP required.

**Security invariants:**
- The full key is never returned by any API response (masked hint only: first 3 + `…` + last 3 chars).
- The full key is never written to any log at any level.
- Audit events `HIBP_API_KEY_UPDATED` and `HIBP_API_KEY_CLEARED` include only the masked hint, not the key value.
- PUT and DELETE require fresh step-up TOTP within the STEPUP_TTL_SECONDS window.

**Database:** new table `auth_settings` (migration `0007_auth_settings.py`). Single row per setting key; values encrypted via pgcrypto `pgp_sym_encrypt` — same mechanism as all other encrypted columns in the schema.

## Changed behaviour

| Area | Before | After |
|---|---|---|
| Password change | No breach check | HIBP k-Anonymity check on every user-chosen password change |
| Password change API response | `400 password_rejected` for all validation failures | `422 password_breached` for HIBP rejections; `400 password_rejected` for length/context failures |
| System-generated passwords | `hash_password()` called without `check_breach` flag | `hash_password(..., check_breach=False)` — system passwords explicitly skip HIBP |

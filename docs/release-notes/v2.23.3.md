## Highlights

### DNS-Rebinding Defence — Pinned Resolver (`feat(security): pinned-resolver vs DNS-rebinding`)

Closes the DNS-rebinding gap in Yashigani's SSRF defences (OWASP API7, issue #91).

Pre-flight allowlist checks in `HttpClient` run before the TCP connection is opened.
An attacker who controls DNS for an allowlisted hostname can serve a safe public IP
on the policy-check DNS query and then swap the record to an internal address (cloud
IMDS, internal services) before the actual connection is made — bypassing the check.

The new `pinned_resolver` context manager closes this race:

1. Resolves the hostname once at context entry using `socket.getaddrinfo`.
2. Verifies the resolved IP against the full SSRF policy (hard-block ranges + operator allowlist/blocklist).
3. Caches the verified IP and patches `socket.getaddrinfo` for the transport layer — so the OS resolver is not called again for that hostname during the connection.
4. Restores the original resolver unconditionally on exit (including exception paths).

The patch scope is narrow: only the exact hostname of the current request URL is intercepted; all other hosts continue to use the real OS resolver. `follow_redirects` is always `False` on the yielded client, preventing redirect-chain bypasses.

Every successful pin emits `SSRF_PINNED_RESOLVER_USED` at DEBUG log level, including the hostname and pinned IP, so operators can confirm the path is active.

See `docs/security/ssrf.md` for full detail on the attack, the defence, remaining
attack surface, and observability guidance.

### Air-gap deployment + supply-chain assurance (`feat(install): air-gap mode #58`)

Yashigani v2.23.3 adds first-class support for installation in isolated
(air-gapped) environments — facilities with no outbound internet access,
classified networks, and regulated environments (CMMC, FedRAMP, NIS2 operators
with strict isolation requirements).

**Customer-builds-the-bundle design.** Agnostic Security ships only a pinned
`airgap/manifest.yml` (kilobytes, committed to the repository). The operator
builds the bundle on their own connected host, pulls from upstream registries
under their own registry credentials and supply-chain attestation tooling, and
transfers the result via whatever out-of-band channel their security policy
permits. Agnostic Security does not hold images, blobs, or signing keys in this
flow — the customer retains full supply-chain control.

**Three-step workflow:**

1. Connected host: `./scripts/prepare-airgap-bundle.sh --profile core` — pulls,
   verifies digests, saves images, compresses into `yashigani-airgap-v2.23.3-core.tar.zst`
   with a SHA256 sidecar manifest.
2. Transfer: bundle + sidecar + `install.sh` + `airgap/manifest.yml` via removable
   media or secure file transfer.
3. Isolated host: `./install.sh --air-gap --bundle yashigani-airgap-v2.23.3-core.tar.zst`

**Digest verification (fail-closed):** Every image digest is verified at pull
time (on the connected host) and again at load time (on the isolated host)
against `airgap/manifest.yml`. A mismatch aborts the install before any service
starts.

**Profile selection:** Operators select which components to bundle:
`core` (always required), `observability` (Grafana/Loki/Prometheus/Jaeger/OTel),
`openwebui`, `agents` (LangFlow/Letta/OpenClaw), `wazuh`, `spire`. The `full`
alias bundles core + observability + Open WebUI.

**Outbound fetch lockout:** `--air-gap` blocks all outbound fetches — registry,
HIBP, ACME, npm, PyPI, GitHub releases. HIBP password check is explicitly
skipped with operator guidance for post-reinstatement rotation.

**Pre-flight gate G20** verifies bundle presence, manifest parseability, zstd
availability, and sidecar presence (warn if absent) before install proceeds.

Full operator guide: [`docs/operations/air-gap-install.md`](../operations/air-gap-install.md)

### PKI Admin UI + BYO-CA Driver (`feat(pki): admin UI + BYO-CA driver`)

Yashigani v2.23.3 adds a full PKI management UI to the backoffice and introduces
the BYO-CA driver, allowing operators to use an external certificate authority
(step-ca, Vault PKI, or any HTTPS signing endpoint) instead of the built-in
self-signed issuer.

**Admin UI endpoints (all require admin session):**

| Method | Path | Auth | Notes |
|---|---|---|---|
| GET | `/api/v1/admin/pki/chain/{service}` | Session | Cert chain detail: CN, SANs, serial, validity, SHA-256 fingerprint |
| POST | `/api/v1/admin/pki/rotate/{service}` | Session + **step-up** | Rotate leaf certificate (ASVS V6.8.4) |
| GET | `/api/v1/admin/pki/bundle/{service}` | Session | Download PEM bundle (leaf + intermediates; private key never included) |
| GET | `/api/v1/admin/pki/status` | Session | All services' cert status: CA mode, needs_renewal flag, fingerprint |

**BYO-CA driver (`YASHIGANI_PKI_CA_MODE=byo`):**
- Default `internal` — existing self-signed issuer; no behaviour change.
- Set `byo` to use an external HTTPS signing endpoint.
- Generates EC P-256 CSR → POSTs to `YASHIGANI_BYO_SIGNING_ENDPOINT` → validates chain → installs atomically.
- Auth modes: `token` (bearer), `mtls` (client cert), `none`.
- Fail-closed: any signing failure raises `DriverError` at startup/rotation; no silent fallback to internal issuer.
- Chain validation: leaf issuer must match BYO CA subject.
- Key write is atomic (`tempfile` + `Path.replace()`); partial state never written.

**Security invariants:**
- Private key is never included in any API response (explicit `PRIVATE KEY` check before streaming bundle).
- Service name validated via regex `[a-z][a-z0-9_\-]{0,63}$` — path traversal prevention.
- Step-up TOTP required for rotation (ASVS V6.8.4).
- Audit events: `PKI_CERT_ROTATED`, `PKI_CERT_ROTATION_FAILED`.

**Environment variables (BYO mode):**

| Variable | Required | Description |
|---|---|---|
| `YASHIGANI_PKI_CA_MODE` | No (default: `internal`) | `internal` or `byo` |
| `YASHIGANI_BYO_SIGNING_ENDPOINT` | Yes (byo) | HTTPS URL of signing endpoint |
| `YASHIGANI_BYO_CA_CERT_PATH` | Yes (byo) | Path to BYO CA certificate (PEM) |
| `YASHIGANI_BYO_SIGNING_AUTH_MODE` | No (default: `token`) | `token`, `mtls`, or `none` |
| `YASHIGANI_BYO_SIGNING_TOKEN` | Cond. (token mode) | Bearer token or `file://path` |
| `YASHIGANI_BYO_CLIENT_CERT_PATH` | Cond. (mtls mode) | Client cert PEM for mTLS to signing endpoint |
| `YASHIGANI_BYO_CLIENT_KEY_PATH` | Cond. (mtls mode) | Client key PEM for mTLS to signing endpoint |

### Licence expiry UX + grace period (Phase 2)

Operators now have explicit, progressive notification of licence expiry across every operator-facing surface. The backoffice banner, a new machine-readable API endpoint, and a deterministic grace-period enforcement model replace the previous silent-fallback-to-Community behaviour.

#### Banner system

Every backoffice admin page shows a licence expiry banner when the expiry is within 30 days. Banner severity escalates automatically:

| Days relative to expiry | Mode | Severity |
|---|---|---|
| >30 days | `active` | Hidden |
| 7–30 days | `warning` | Yellow — "Yashigani licence expires in X days — renew to avoid service interruption." |
| 1–7 days | `critical` | Orange — "...renew now, sales@agnosticsec.com." |
| 0 days (expires today) | `critical` | Orange |
| 0–14 days past expiry | `expired` | Red — grace period remaining shown |
| 14–30 days past | `readonly` | Red — "Gateway is in read-only mode" |
| 30+ days past | `blocked` | Red — "Gateway is blocked" |

#### Grace-period enforcement

Post-expiry, Yashigani degrades service progressively instead of cutting off immediately:

- **Days 0–14 past expiry (EXPIRED / grace):** full service continues. Red banner on all pages. Daily audit log entry written.
- **Day 14–30 (READONLY):** operators can view all admin pages but cannot change configuration; new agent-runs are blocked. In-flight agent runs complete. HTTP 403 returned for write operations.
- **Day 30+ (BLOCKED):** HTTP 503 returned on all data-plane requests with `Retry-After` header and renewal URL.

This is a fail-explicit, not fail-silent, design: each mode is logged, audited, and surfaced to the operator, consistent with `feedback_zero_trust_default.md`.

#### Machine-readable expiry status API

New endpoint available to monitoring scripts and CLI tools:

```
GET /api/v1/license/status          (also at GET /admin/license/status)
Authorization: admin session cookie

Response:
{
  "valid": true,
  "expires_at": "2026-12-01T00:00:00+00:00",
  "days_remaining": 208,
  "grace_period_active": false,
  "mode": "active"
}
```

`mode` values: `"active"` | `"warning"` | `"critical"` | `"expired"` | `"readonly"` | `"blocked"`.

Authentication required (admin session). Accessible in READONLY mode (read-only-safe path).

#### Daily grace-period audit events

When in `expired`, `readonly`, or `blocked` mode, the scheduler emits one audit event per calendar day with:
- `event_type: "LICENSE_GRACE_PERIOD"`
- `component` field: mode, days-since-expiry, expires_at

Provides an auditable trail for GRC / compliance purposes without flooding logs.

## Security

### Secret rotation API (ASVS V6.8.4, V7.1.2, V12.4.1)

**Finding addressed:** Secrets were only rotatable via full nuke-and-reinstall
(`install.sh`). This created operational pressure to avoid rotation, increasing
exposure windows.

**Change:** New endpoint `POST /api/v1/admin/secrets/rotate` with:
- Step-up TOTP required (ASVS V6.8.4 re-authentication before critical operations).
- Fail-closed: if rotation fails mid-way, the old secret is reverted. If revert
  also fails, the event is logged `CRITICAL` and the admin is alerted.
- Atomic secret file writes: temp file + `rename(2)` — concurrent readers never
  see a partial secret.
- Immutable audit trail: `SECRET_ROTATION_REQUESTED` → `SECRET_ROTATION_SUCCEEDED`
  or `SECRET_ROTATION_FAILED` in the SHA-384 audit chain. Secret values are never
  stored in audit events.
- Body limit: 256 bytes (ASVS 4.3.1).

**Password charset:** New passwords use `A-Za-z0-9!*,-._~` per
`feedback_password_charset.md` — URL-safe (no `%` encoding in DSNs), shell-safe
(no `$`, backtick, `|`, `&`, `;` in double-quoted strings), 48 characters.

**CLI wrapper:** `scripts/rotate-secret.sh <secret-name>` wraps the API and
runs `docker/podman compose restart` for the affected services from the host.

**Documentation:** See `docs/yashigani_install_config.md §28` for full
operational runbook including failure recovery for the `revert_failed=true` case.

### ACS product gaps — 5 controls closed (#95)

Five gaps identified by ACS v3.0 scan against Yashigani v2.23.3-dev are closed in this release.

#### Gap 1 — Auth audit log completeness (CMMC AU.L2-3.3.1 / OWASP ASVS V7.2.1)

Four new `EventType` variants added to `audit/schema.py`:

| EventType | Emitted when |
|---|---|
| `AUTH_LOGIN_ATTEMPT` | Admin login route reached; before credentials are checked |
| `ACCOUNT_LOCKOUT` | `pg_auth.authenticate()` detects password or TOTP failure count at threshold |
| `PASSWORD_CHANGED` | Admin password changed; includes `change_type`, `old_hash_tail`, `new_hash_tail` (masked, not full hashes), `sessions_invalidated` flag |
| `SESSIONS_INVALIDATED` | Admin sessions invalidated; includes `acting_admin`, `reason`, `sessions_count` |

`AUTH_LOGIN_ATTEMPT` events mask the last IPv4 octet (`a.b.c.0`). `ACCOUNT_LOCKOUT` events are emitted by a non-blocking helper — exceptions are swallowed with a WARNING log so the auth response is never delayed.

#### Gap 2 — SCIM filter injection (OWASP API3 / CWE-20)

`GET /api/v1/admin/scim/v2/Users?filter=` now uses `Query(max_length=256)` — FastAPI enforces the length limit before any Python code runs. `_parse_filter_email()` validates candidates against `_EMAIL_RE` (RFC 5321-shaped), rejecting injection attempts.

#### Gap 3 — BFLA on service management (ASVS V6.8.4 / CMMC AC.L2-3.1.1)

`POST /api/v1/admin/services/{service_id}` dependency changed from `AdminSession` to `StepUpAdminSession`. A fresh TOTP step-up is now required before enabling or disabling a compose-profile service. `GET` (list) retains `AdminSession`.

#### Gap 4 — Third-party response validation (OWASP API8 / CWE-20)

**HIBP:** lines now validated against `^[0-9A-F]{35}:[0-9]+$` before `int()` conversion; malformed lines logged and skipped.

**OIDC discovery:** metadata validated by `_validate_oidc_metadata()` immediately after fetch — `issuer`, `authorization_endpoint`, `token_endpoint`, `jwks_uri` must be present, non-empty, and `https://`-prefixed before the document is cached.

#### Gap 5 — CMMC AC.L2-3.1.1 least-privilege

Same root cause as Gap 3; closed by the same `StepUpAdminSession` fix.

**Test coverage:** 65 new unit tests in `src/tests/unit/test_v2233_acs_product_gaps.py`.

### ASVS V2.1.7 — HIBP k-Anonymity breach check

**OWASP ASVS V2.1.7:** "Verify that passwords submitted during account registration, login, and password change are checked against a set of breached passwords."

**Implementation:**

1. On every user-chosen password change, the proposed password is SHA-1 hashed.
2. The first 5 hex characters of the hash are sent to `api.pwnedpasswords.com/range/{prefix}` (k-Anonymity: only the prefix, never the full hash or password, is transmitted).
3. The API response contains all hash suffixes matching that prefix. If the full hash suffix is found, the password has appeared in a known breach.
4. A matching password is rejected with **HTTP 422 Unprocessable Entity** and the message: _"This password has appeared in known data breaches; choose another."_
5. Non-matching passwords proceed to Argon2id hashing and storage as normal.

**Scope:** user-chosen passwords only. System-generated temporary passwords (created at account bootstrap, admin-forced reset, or full account reset) are exempt — they are randomised with a 36-char secure alphabet and are not user-chosen.

**Failure mode (fail-open):** if the HIBP API is unreachable (network timeout, 5xx, DNS failure), the password change is allowed to proceed. A `WARNING` log entry is emitted and the `yashigani_hibp_check_api_unavailable_total` Prometheus counter is incremented. This preserves operator access during API outages.

**Privacy:** the full password and full SHA-1 hash are never transmitted. Only the first 5 characters of the SHA-1 hex digest leave the system. SHA-1 is used because it is mandated by the HIBP k-Anonymity wire protocol (NIST SP 800-63B §5.1.1.2); it is not used as a storage primitive (passwords are stored with Argon2id).

**Operator controls:**

| Environment variable | Default | Description |
|---|---|---|
| `YASHIGANI_HIBP_CHECK_ENABLED` | `true` | Set to `false` to disable HIBP checks. For air-gapped deployments. Disabling is a risk-accept against ASVS V2.1.7 — document in your risk register. |
| `YASHIGANI_HIBP_API_URL` | `https://api.pwnedpasswords.com/range/` | Override the API base URL to point at an internal HIBP mirror. Include trailing slash. |
| `YASHIGANI_HIBP_API_KEY` | _(none)_ | Optional API key for commercial-tier rate-limit lift or mirror auth. Prefer admin-panel configuration (see below). |

**Network requirement:** `backoffice` container requires outbound HTTPS to `api.pwnedpasswords.com:443` on the `edge` network (default topology). Use `YASHIGANI_HIBP_API_URL` to redirect to an internal mirror for air-gapped deployments.

**Observability:** the `yashigani_hibp_check_api_unavailable_total` Prometheus counter tracks failed HIBP API calls (each failure = one password change where the breach check was skipped due to API unreachability). Alert on sustained non-zero rate to detect loss of outbound edge connectivity.

### Admin-panel HIBP API key (v2.23.3 extension to PR #59)

Operators who need the HIBP commercial-tier rate-limit lift, or who run a self-hosted HIBP mirror requiring authentication, can now set and rotate the HIBP API key from the admin panel — **no container restart required**.

**Admin UI location:** Settings → Authentication > HIBP API Key

**Resolution priority (highest to lowest):**
1. Admin-panel key — encrypted at rest with the deployment AES key (`YASHIGANI_DB_AES_KEY`); takes effect immediately.
2. Env var `YASHIGANI_HIBP_API_KEY` — fallback for automation.
3. None — anonymous k-Anonymity request (free tier; works for most deployments).

**API endpoints:**
- `GET /api/v1/admin/auth/hibp/status` — returns `{configured, source, masked_value, updated_at, updated_by}`. Admin session required. Full key never returned.
- `PUT /api/v1/admin/auth/hibp/key` — set key. Body: `{api_key: "..."}`. Step-up TOTP required (ASVS V6.8.4).
- `DELETE /api/v1/admin/auth/hibp/key` — clear key. Step-up TOTP required.

**Security invariants:**
- The full key is never returned by any API response (masked hint only: first 3 + `…` + last 3 chars).
- The full key is never written to any log at any level.
- Audit events `HIBP_API_KEY_UPDATED` and `HIBP_API_KEY_CLEARED` include only the masked hint, not the key value.
- PUT and DELETE require fresh step-up TOTP within the STEPUP_TTL_SECONDS window.

**Database:** new table `auth_settings` (migration `0008_auth_settings.py`). Single row per setting key; values encrypted via pgcrypto `pgp_sym_encrypt` — same mechanism as all other encrypted columns in the schema.

### Security

- **Password reuse history (CMMC L2 IA.L2-3.5.8):** password changes are now
  checked against the last `PASSWORD_HISTORY_DEPTH` (default 12) Argon2id hashes
  stored in the new `password_history` table. Reuse is rejected with HTTP 422
  error code `password_reuse`. Emits `PASSWORD_REUSE_REJECTED` audit event with
  `user_id` and `history_depth_checked` — no password or hash stored in event.
  History is pruned to `depth` entries after each successful change.
  See `docs/security/authentication.md` §Password Reuse History for full details.
- sign_count rollback detection: new_sign_count ≤ stored (when stored > 0) is

  `no_credentials_registered` 400 as a known user with no keys.
- Cross-origin attestation rejected by `py_webauthn` via `expected_origin`
  parameter derived from `X-Forwarded-Proto` + `Host` headers.

#### F-V233-002 — age-encrypted backups (MP.L2-3.8.9 / CMMC L2)

Closes the CMMC L2 product gap for backup media protection (MP.L2-3.8.9 /
CWE-312 cleartext storage at rest).

- **`scripts/backup.sh`** (new): produces `<timestamp>.tar.gz.age` via a
  `tar -cz ... | age -r <recipient>` pipeline. Recipient public key read from
  `/etc/yashigani/backup-recipient.age.pub` (operator-provisioned). Fails
  loudly with operator remediation steps if the key file is absent or
  malformed (starts with `age1` — bech32 X25519 format). Binary output only
  (not ASCII-armored). Temp file created atomically then renamed; cleaned up
  on SIGINT/SIGTERM. Output mode `0400`, output dir `0700`.

- **`restore.sh`** (updated): `--encrypted <identity.age> <archive.tar.gz.age>`
  decrypts with the operator's age identity then delegates to the existing
  restore flow. Legacy unencrypted `.tar.gz` archives are still accepted with
  a deprecation warning. Auto-detects encrypted archives by `.tar.gz.age`
  suffix even without the `--encrypted` flag (when
  `YASHIGANI_BACKUP_IDENTITY_FILE` is set). Identity file permission check
  warns on world/group-readable modes (CWE-732).

- **`docker/Dockerfile.backoffice` + `docker/Dockerfile.gateway`** (updated):
  `apt-get install -y age=1.2.1-1+b5` added to runtime stage. Version pinned
  to Debian bookworm available version per image-pinning rule.

- **Helm chart** (updated): `backup.encryption.recipientKeyConfigMap` +
  `backup.encryption.identitySecret` values added. New `backup-cronjob.yaml`
  template: scheduled CronJob mounting the ConfigMap (public key) for
  encryption-only operation. New `backup-script` ConfigMap embeds
  `backup.sh`. CronJob does NOT mount the identity Secret (encryption is
  public-key only). PVC support via `backup.pvcName` or `backup.pvc.create`.

- **`scripts/preflight.sh`** (updated): Gate G19 added — checks `age` binary
  present in PATH (WARN) and recipient key file readable + starts with `age1`
  (FAIL if file exists but is malformed).

- **`docs/operations/backup.md`** (new): operator key generation steps,
  recipient extraction, Kubernetes setup, retention, and key rotation runbook.

**Encryption:** AES-256-GCM via age X25519 key exchange. No symmetric shared
secret stored on the backup server.

**Restore round-trip:** Docker, Podman (rootless), K8s all supported via
`restore.sh --encrypted`.

## K8s rolling-restart note for `hmac_key` (W10)

When `hmac_key` is rotated on a K8s deployment with multiple Caddy/gateway pods,
the ~1-2 request inconsistency window documented for Compose deployments becomes
a per-pod rolling-restart window.  Each pod restarts sequentially (controlled by
Helm `strategy.rollingUpdate.maxUnavailable`); during the rollout, pods that have
not yet restarted hold the old HMAC key while new pods use the rotated key.
This is bounded by the rollout duration, not by request count.

**Runbook for K8s `hmac_key` rotation:**
1. Call `POST /api/v1/admin/secrets/rotate` (step-up TOTP required).
2. Immediately trigger a rolling restart: `kubectl rollout restart deployment/yashigani-caddy deployment/yashigani-gateway deployment/yashigani-backoffice`.
3. Monitor: `kubectl rollout status deployment/yashigani-caddy` until complete.
4. The inconsistency window is bounded by `maxUnavailable * podStartupSeconds`.

## Audit chain note for `SECRET_ROTATION_REQUESTED` (W11)

`SECRET_ROTATION_REQUESTED` is emitted before the rotation begins (intentional —
it records the operator intent even if the service crashes mid-rotation).

**Operators and GRC tools**: if the audit chain shows `SECRET_ROTATION_REQUESTED`
without a paired `SECRET_ROTATION_SUCCEEDED` or `SECRET_ROTATION_FAILED` event,
this indicates the backoffice service crashed during rotation.  The recovery
procedure is:

1. Check container logs for the crash cause.
2. Verify which secrets were successfully rotated (audit events before the crash,
   container filesystem, Postgres `app_setting` table).
3. Rotate any secrets that may be in an inconsistent state before restarting.

This is documented in `docs/yashigani_install_config.md §28.8 (Recovery Runbook)`.

## Upgrade Notes

1. Run `alembic upgrade head` (or restart the stack — the lifespan runs
   `run_migrations()` automatically).
2. Migration `0007` alters the `webauthn_credentials` table — no data loss,
   all existing rows are preserved.
3. Set `YASHIGANI_WEBAUTHN_RP_ID` to your domain before the first registration.
4. TLS is required for WebAuthn — ensure Caddy is configured correctly.
5. **Remove `YASHIGANI_WEBAUTHN_ORIGIN` from your `.env` file** — this variable
   is no longer read. The WebAuthn origin is now derived automatically from the
   `X-Forwarded-Proto` and `Host` headers on each request (set by Caddy). If you
   leave the variable set it will be silently ignored; removing it avoids operator
   confusion on future upgrades. Previous variables `YASHIGANI_WEBAUTHN_RP_ID` and
   `YASHIGANI_WEBAUTHN_RP_NAME` remain required and unchanged.

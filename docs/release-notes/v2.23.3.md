## Highlights

### Admin-triggered secret rotation (`feat(secrets): rotate-secrets`)

Yashigani v2.23.3 introduces zero-downtime, admin-triggered rotation of
infrastructure credentials via the backoffice API. No nuke-and-rebuild required.

**Supported secrets:**

| Secret | Mechanism | Zero-downtime? |
|---|---|---|
| `postgres_password` | `ALTER USER` via direct Postgres connection; pgbouncer restart | Yes (dual-credential window) |
| `redis_password` | `CONFIG SET requirepass` in-flight | Yes |
| `jwt_signing_key` | File hot-swap; existing tokens valid until expiry | Yes |
| `hmac_key` | File hot-swap; brief ~1-2 request window on reload (Compose/Podman); K8s: window is per-pod restart, not bounded to 1-2 requests — see K8s note below | Near-zero (Compose); bounded per-pod (K8s) |
| `all` | Sequential rotation of all four | Inherits per-secret |

### Licence expiry UX + grace period (Phase 2)

Operators now have explicit, progressive notification of licence expiry across every operator-facing surface. The backoffice banner, a new machine-readable API endpoint, and a deterministic grace-period enforcement model replace the previous silent-fallback-to-Community behaviour.

#### Banner system

Every backoffice admin page shows a licence expiry banner when the expiry is within 30 days. Banner severity escalates automatically:

| Days relative to expiry | Mode | Severity |
|---|---|---|
| >30 days | `active` | Hidden |
| 7–30 days | `warning` | Yellow — "Yashigani licence expires in X days — renew to avoid service interruption." |
| 1–7 days | `critical` | Orange — "...renew now, sales@agnosticsec.com." |
| 0 days (expires today) | `critical` | Orange |
| 0–14 days past expiry | `expired` | Red — grace period remaining shown |
| 14–30 days past | `readonly` | Red — "Gateway is in read-only mode" |
| 30+ days past | `blocked` | Red — "Gateway is blocked" |

#### Grace-period enforcement

Post-expiry, Yashigani degrades service progressively instead of cutting off immediately:

- **Days 0–14 past expiry (EXPIRED / grace):** full service continues. Red banner on all pages. Daily audit log entry written.
- **Day 14–30 (READONLY):** operators can view all admin pages but cannot change configuration; new agent-runs are blocked. In-flight agent runs complete. HTTP 403 returned for write operations.
- **Day 30+ (BLOCKED):** HTTP 503 returned on all data-plane requests with `Retry-After` header and renewal URL.

This is a fail-explicit, not fail-silent, design: each mode is logged, audited, and surfaced to the operator, consistent with `feedback_zero_trust_default.md`.

#### Machine-readable expiry status API

New endpoint available to monitoring scripts and CLI tools:

```
GET /api/v1/license/status          (also at GET /admin/license/status)
Authorization: admin session cookie

Response:
{
  "valid": true,
  "expires_at": "2026-12-01T00:00:00+00:00",
  "days_remaining": 208,
  "grace_period_active": false,
  "mode": "active"
}
```

`mode` values: `"active"` | `"warning"` | `"critical"` | `"expired"` | `"readonly"` | `"blocked"`.

Authentication required (admin session). Accessible in READONLY mode (read-only-safe path).

#### Daily grace-period audit events

When in `expired`, `readonly`, or `blocked` mode, the scheduler emits one audit event per calendar day with:
- `event_type: "LICENSE_GRACE_PERIOD"`
- `component` field: mode, days-since-expiry, expires_at

Provides an auditable trail for GRC / compliance purposes without flooding logs.

## Security

### Secret rotation API (ASVS V6.8.4, V7.1.2, V12.4.1)

**Finding addressed:** Secrets were only rotatable via full nuke-and-reinstall
(`install.sh`). This created operational pressure to avoid rotation, increasing
exposure windows.

**Change:** New endpoint `POST /api/v1/admin/secrets/rotate` with:
- Step-up TOTP required (ASVS V6.8.4 re-authentication before critical operations).
- Fail-closed: if rotation fails mid-way, the old secret is reverted. If revert
  also fails, the event is logged `CRITICAL` and the admin is alerted.
- Atomic secret file writes: temp file + `rename(2)` — concurrent readers never
  see a partial secret.
- Immutable audit trail: `SECRET_ROTATION_REQUESTED` → `SECRET_ROTATION_SUCCEEDED`
  or `SECRET_ROTATION_FAILED` in the SHA-384 audit chain. Secret values are never
  stored in audit events.
- Body limit: 256 bytes (ASVS 4.3.1).

**Password charset:** New passwords use `A-Za-z0-9!*,-._~` per
`feedback_password_charset.md` — URL-safe (no `%` encoding in DSNs), shell-safe
(no `$`, backtick, `|`, `&`, `;` in double-quoted strings), 48 characters.

**CLI wrapper:** `scripts/rotate-secret.sh <secret-name>` wraps the API and
runs `docker/podman compose restart` for the affected services from the host.

**Documentation:** See `docs/yashigani_install_config.md §28` for full
operational runbook including failure recovery for the `revert_failed=true` case.

## K8s rolling-restart note for `hmac_key` (W10)

When `hmac_key` is rotated on a K8s deployment with multiple Caddy/gateway pods,
the ~1-2 request inconsistency window documented for Compose deployments becomes
a per-pod rolling-restart window.  Each pod restarts sequentially (controlled by
Helm `strategy.rollingUpdate.maxUnavailable`); during the rollout, pods that have
not yet restarted hold the old HMAC key while new pods use the rotated key.
This is bounded by the rollout duration, not by request count.

**Runbook for K8s `hmac_key` rotation:**
1. Call `POST /api/v1/admin/secrets/rotate` (step-up TOTP required).
2. Immediately trigger a rolling restart: `kubectl rollout restart deployment/yashigani-caddy deployment/yashigani-gateway deployment/yashigani-backoffice`.
3. Monitor: `kubectl rollout status deployment/yashigani-caddy` until complete.
4. The inconsistency window is bounded by `maxUnavailable * podStartupSeconds`.

## Audit chain note for `SECRET_ROTATION_REQUESTED` (W11)

`SECRET_ROTATION_REQUESTED` is emitted before the rotation begins (intentional —
it records the operator intent even if the service crashes mid-rotation).

**Operators and GRC tools**: if the audit chain shows `SECRET_ROTATION_REQUESTED`
without a paired `SECRET_ROTATION_SUCCEEDED` or `SECRET_ROTATION_FAILED` event,
this indicates the backoffice service crashed during rotation.  The recovery
procedure is:

1. Check container logs for the crash cause.
2. Verify which secrets were successfully rotated (audit events before the crash,
   container filesystem, Postgres `app_setting` table).
3. Rotate any secrets that may be in an inconsistent state before restarting.

This is documented in `docs/yashigani_install_config.md §28.8 (Recovery Runbook)`.

## Highlights

### OWASP Agentic AI Top 10 — Overreliance controls (F-T10-001)

The load-bearing security feature of this release. Every LLM and agent response routed through the gateway now carries three operator-visible signals that surface AI uncertainty and prevent uncritical trust in generated output, directly addressing OWASP Agentic AI Top 10 item: Overreliance.

Three response headers are injected on both the buffered and streaming paths (see [Security](#security) below for full detail).

### Multi-arch image pin sweep — P-8 systemic close (#43)

All external container images are now pinned to multi-arch manifest INDEX digests (`application/vnd.oci.image.index.v1+json`) rather than arch-specific layer digests. Root cause: `jaeger:2.17.0` was pinned to an `linux/amd64`-specific manifest SHA, causing exec-format errors and a podman-compose sequential-block on aarch64 runners. The sweep replaces every `@sha256:` pin across `docker-compose.yml`, `docker-compose.release.yml`, Helm chart values, and all related configuration files. Closes Podman parity registry finding P-8 and smoke-test gate V232-SMOKE-001.

### Podman healthcheck parity — P-9 close (#52)

Silent healthcheck drop on Podman rootless removed. See [Operational](#operational) below. Merge commit `10864bd`.

### Non-interactive install and uninstall (#46, #47)

`install.sh` now auto-creates bind-mount directories for all runtimes before starting the stack, eliminating a class of startup race on Docker rootful installs. `uninstall.sh` gains `--yes` / `-y` flags for unattended removal in automation and CI pipelines.

## Security

### F-T10-001 — OWASP Agentic AI T10 Overreliance UX controls

Three response headers are added to every LLM/agent response on both the OpenAI-compatible (`/v1/*`) and catch-all MCP/tool proxy paths:

| Header | Value | Purpose |
|--------|-------|---------|
| `X-Yashigani-Generated-Content` | `true` | Always present. Enables operator UIs to render AI-generated badges or disclosure banners without heuristic detection. |
| `X-Yashigani-Response-Inspection-Confidence` | `<float [0.0–1.0]>` | Inspection pipeline confidence score. Defaults to `1.0000` when inspection is disabled or skipped (clean-pass). |
| `X-Yashigani-Low-Confidence-Stepup` | `required` | Emitted when inspection confidence is below `YASHIGANI_LOW_CONFIDENCE_STEPUP_THRESHOLD` (default `0.50`) AND the request `sensitivity_level` is `CONFIDENTIAL` or `RESTRICTED`. Signals the operator UI to surface a "verify before acting" prompt. |

**Streaming path:** `X-Yashigani-Low-Confidence-Stepup` is intentionally omitted on streamed responses — confidence is not computable pre-stream; the `StreamingInspector` handles in-band signalling.

**Confidence clamping:** The confidence value is clamped to `[0.0, 1.0]` via `math.isfinite()` before header formatting, preventing `NaN` or `Inf` from reaching the wire.

**Env-var fallback:** `YASHIGANI_LOW_CONFIDENCE_STEPUP_THRESHOLD` accepts a float string; defaults to `0.50` on parse error or when the var is unset.

16 regression tests in `src/tests/unit/test_f_t10_001_overreliance.py` assert header presence, confidence defaults, stepup threshold logic, sensitivity guards, and streaming path behaviour.

**OWASP references:** Agentic AI Top 10 (Overreliance); ASVS V13.2.6 (LLM output handling).

## Operational

### Non-interactive install (#47)

`install.sh` pre-creates all bind-mount directories before invoking `docker compose up` / `podman-compose up`. This eliminates a class of startup race on Docker rootful installs where the daemon creates mount directories as `root:root 755`, preventing the container user from writing. No sudo is invoked during the installer body (pre-flight may sudo; installer body is sudo-free per [install audience SOP](../yashigani_install_config.md)).

### Non-interactive uninstall (#46)

`uninstall.sh` accepts `--yes` / `-y` to skip the interactive confirmation prompt. Intended for use in CI teardown, automated test pipelines, and configuration-management tooling.

```sh
# Example: unattended removal
./uninstall.sh --yes
```

### Podman healthcheck parity (#52)

Silent healthcheck drop on Podman rootless (Podman parity registry P-9) is resolved. The fix ensures that the container `HEALTHCHECK` command is wired consistently for both Docker and Podman runtimes and that health-state transitions are observable via `podman inspect`. Merge commit `10864bd`.

### Grafana 13.0.1 (#45)

Grafana image bumped from `12.4.3` to `13.0.1` across `docker-compose.release.yml`, `helm/charts/grafana/values.yaml`, `helm/yashigani/values.release.yaml`, and `helm/yashigani/values.yaml`. The `helm/charts/grafana/values.yaml` "latest" pinning gap (Podman parity registry `latest`-tag risk) is simultaneously closed: the image is now pinned to a specific version tag in all four files.

## Breaking Changes

### Rate limiter fail-closed default (F-LLM06-001 / F-V232-001)

**`RATE_LIMITER_FAIL_MODE` now defaults to `closed`.**

In v2.23.1 and earlier, when Redis became temporarily unreachable, the rate limiter silently allowed all traffic through (`fail_mode=open`). For a security gateway this is the wrong default: a Redis blip removed all rate-limit enforcement, exposing the LLM backend to unbounded traffic without logging or throttling.

From v2.23.2, the default is `closed`. When Redis is unavailable, the gateway returns:

```
HTTP 503 Service Unavailable
Retry-After: 5
Content-Type: application/json

{"error": "RATE_LIMITER_UNAVAILABLE", "retry_after_seconds": 5}
```

Auto-heal is typically complete in under 2 minutes. This behaviour aligns with the zero-trust secure-by-default posture (`feedback_zero_trust_default.md`) and OWASP LLM Top 10 LLM06 (Excessive Agency) finding F-LLM06-001.

**Operator opt-out (high-availability deployments):** operators who require pre-2.23.2 fail-open behaviour — for example, because a WAF, per-IP throttle, or other coarser control is sufficient during brief Redis instability — can opt back in explicitly:

```sh
# docker-compose.yml / helm values / environment block
RATE_LIMITER_FAIL_MODE=open
```

Setting this variable in the gateway service environment restores v2.23.1 behaviour. The value is validated at startup; invalid values log a warning and fall back to `closed`.

The only other observable change at the API surface is the addition of the three `X-Yashigani-*` response headers on LLM/agent responses (see [Security: F-T10-001](#f-t10-001--owasp-agentic-ai-t10-overreliance-ux-controls) above). Operator UIs that do not consume these headers will not be affected.

## Upgrade Path

### In-place upgrade (code-only change)

```sh
# Pull updated images (image pins changed in #43 — fresh pull required)
docker compose pull   # or: podman-compose pull

# Restart the stack
docker compose up -d  # or: podman-compose up -d
```

No database migrations in this release. No new secrets required.

### Clean re-install

Follow the standard install SOP from [`docs/yashigani_install_config.md`](../yashigani_install_config.md). Backup before nuking the existing stack:

```sh
./install.sh backup
./uninstall.sh --yes
./install.sh [podman|docker|k8s]
./install.sh restore <backup-file>
```

### Kubernetes / Helm

```sh
helm upgrade yashigani helm/yashigani/ -f helm/yashigani/values.release.yaml
```

Image pins in `values.release.yaml` have been updated to INDEX digests (#43). Run `helm diff upgrade` first to preview the change set.

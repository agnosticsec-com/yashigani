AGENT BUNDLES — notes for operators:

  ARM64 (Apple Silicon / Graviton): The default agent-bundle image tags are
  multi-arch (linux/amd64 + linux/arm64). If you see ImagePullBackOff with
  "no match for platform", your cluster node is ARM64 and the cached image
  index does not include an ARM64 manifest for that tag. Override at install:
    --set agentBundles.langflow.image.repository=docker.io/langflowai/langflow
    --set agentBundles.langflow.image.tag=1.9.2
    --set agentBundles.letta.image.repository=docker.io/letta/letta
    --set agentBundles.letta.image.tag=0.16.7
  If the platform error persists, use `docker buildx imagetools inspect
  <image>:<tag>` to confirm ARM64 manifest presence before enabling the bundle.
  Finding: HELM-K8S-ARM64-IMAGE-DEFAULT (secondary, 2026-05-13).

  Kyverno admission policies (admissionPolicies.enabled) default to FALSE.
  To enable, install Kyverno cluster-wide first:
    helm repo add kyverno https://kyverno.github.io/kyverno/
    helm repo update
    helm install kyverno kyverno/kyverno -n kyverno --create-namespace
  Then deploy Yashigani with --set admissionPolicies.enabled=true.
  The chart will fail-loud with remediation steps if Kyverno CRDs are absent.
  Finding: ccc7abf/adf6fda (2026-05-15).

AUDIT LOG — DURABILITY NOTICE (Iris coord #3 / Lu A.8.15):

  auditLog.enabled defaults to FALSE. In this state the gateway and backoffice
  write audit events to an emptyDir volume (50 MiB, NOT persistent). Pod restarts
  wipe the audit trail. This is a durability gap vs the Docker/Podman compose
  deployment where audit logging is always-on to a bind-mounted host directory.

  To enable persistent audit logging on Kubernetes:
    --set auditLog.enabled=true
    --set auditLog.persistence.size=5Gi          # adjust to retention needs
    --set auditLog.persistence.storageClass=...  # set for your cluster StorageClass

  ROLLBACK WARNING: if you previously ran with auditLog.enabled=true and
  switch back to enabled=false, the audit PVC becomes orphaned (Helm does not
  delete PVCs on upgrade). Manually delete it after archiving the audit data:
    kubectl delete pvc -n {{ .Release.Namespace }} -l app.kubernetes.io/component=audit-log

  COMPLIANCE NOTE: SOC 2 CC7.2, ISO 27001 A.8.15, NIST AU-9/AU-11, CMMC
  AU.L2-3.3.4/8/9, EU AI Act Art.12 all require durable audit records.
  auditLog.enabled=false does NOT satisfy these controls — document the gap
  in your risk register if running without persistent audit.

MCP BUNDLE (agentBundles.filesystemMcp) — OPERATOR RUNBOOK (K8s SB-2):

  The filesystem-mcp bundle requires a JWT signing key delivered as a K8s
  Secret. In Docker/Compose, install.sh writes the key to a host path and
  bind-mounts it. On Kubernetes, you must pre-create the Secret:

    kubectl create secret generic yashigani-mcp-signing-key \
      -n {{ .Release.Namespace }} \
      --from-literal=mcp_identity_signing_key="$(openssl rand -hex 32)"

  Then reference it in your values override:

    gateway:
      mcpSigningKey:
        existingSecretName: yashigani-mcp-signing-key
        secretKey: mcp_identity_signing_key

  IMPORTANT: gateway.mcpServers must NOT be set unless
  agentBundles.filesystemMcp.enabled=true — the chart will fail at template
  time if this invariant is violated (validate-security.yaml MCP-GUARD-001).
  Set BOTH or NEITHER.

  For GitOps/Sealed Secrets workflows, seal the Secret with:
    kubeseal --scope namespace-wide -o yaml \
      < mcp-signing-key.yaml > sealed-mcp-signing-key.yaml
  and commit the SealedSecret to your cluster repo.

KUBERNETES vs DOCKER/PODMAN FOOTPRINT DIFFERENCES (B11 — DRIFT-001):

  The following components are present in the Docker/Podman Compose deployment
  but are NOT available in this Helm chart. Compliance controls that reference
  these components MUST note this Kubernetes limitation:

  | Component           | Compose | Helm/K8s | Notes                               |
  |---------------------|---------|----------|-------------------------------------|
  | Wazuh SIEM          | YES     | NO       | Deploy external Wazuh + forward logs|
  | Keycloak IdP        | NO      | NO       | Future roadmap item                 |
  | HashiCorp Vault     | NO      | NO       | Use external-secrets-operator       |
  | step-ca (ACME CA)   | NO      | NO       | Use cert-manager instead            |
  | Promtail log agent  | NO      | NO       | Use Loki's native K8s log discovery |
  | OpenClaw UI (full)  | Partial | Partial  | API mode only on both paths         |

  wazuh.enabled=true is explicitly blocked by this chart — it has no effect and
  will fail helm install/upgrade with a configuration error to prevent false
  compliance posture. See values.yaml wazuh section for migration notes.

# ringfence-init — L1 Network-Plane Containment Sidecar

Plan reference: `p1-universal-ringfence-implementation-plan-20260528.md` §2.D L1–L8 + CRIT-01/02/03.

## What it does

`ringfence-init` is a minimal init-sidecar that runs in the agent's network namespace and applies a `default-deny OUTPUT` iptables policy, permitting only one egress path: the resolved Caddy IP on port 443.

Enforcement model: **Layer 2 — network plane**. The agent container has no direct outbound. Caddy is the sole reachable peer. Every outbound call traverses: `Agent → Caddy (mTLS, SPIFFE-verified) → OPA → external`.

## Expected runtime behaviour

### Compose path

```yaml
# codegen output (docker-compose.override.yml snippet):
ringfence-init-<agent>:
  image: ghcr.io/agnosticsec/ringfence-init:0.1.0@sha256:<digest>
  network_mode: "service:<agent>"
  cap_drop: [ALL]
  cap_add: [NET_ADMIN]
  security_opt: [no-new-privileges:true]
  environment:
    RINGFENCE_CADDY_HOST: caddy
    RINGFENCE_CADDY_PORT: "443"
    RINGFENCE_DNS_SERVER: "127.0.0.11"
    RINGFENCE_RUNTIME: "${YSG_RUNTIME}"   # injected by codegen
    RINGFENCE_AGENT_NAME: "<agent>"
  tmpfs:
    - /run/ringfence:size=1m,mode=1777
  # NOTE: network_mode: service:<agent> means this container shares the
  # agent's network namespace. iptables rules written here apply to the
  # agent container (they share one netns).

<agent>:
  ...
  depends_on:
    ringfence-init-<agent>:
      condition: service_completed_successfully  # Compose v2.3+
  networks:
    - ringfence_<agent>     # internal: true, enable_ipv6: false
    - caddy_internal        # so Caddy can reach the agent inbound
```

Sequence:
1. `ringfence-init-<agent>` starts, shares `<agent>`'s network namespace.
2. Resolves `caddy` → Caddy container IP (DNS works before DROP is applied).
3. Applies `iptables OUTPUT DROP` + `ACCEPT to <caddy-ip>:443` + `ACCEPT DNS`.
4. Writes `/run/ringfence/ready`.
5. Exits 0.
6. `<agent>` starts (depends_on `service_completed_successfully`).
7. Agent network namespace now has L1 enforcement: only Caddy-egress allowed.

### K8s path

On K8s the init-sidecar runs as a Kubernetes `initContainers` entry. The Kyverno `ClusterPolicy` drops `NET_ADMIN` from all containers; a generated `PolicyException` scoped to `yashigani.io/ringfence-init: "true"` exempts this sidecar.

```yaml
# templates/agents/<agent>-policy-exception.yaml (generated by codegen)
apiVersion: kyverno.io/v1
kind: PolicyException
metadata:
  name: ringfence-init-net-admin-<agent>
  namespace: yashigani
spec:
  exceptions:
    - policyName: restrict-capabilities
      ruleNames: [require-drop-all]
  match:
    any:
      - resources:
          kinds: [Pod]
          namespaces: [yashigani]
          selector:
            matchLabels:
              yashigani.io/ringfence-init: "true"
              yashigani.io/agent: <agent>
```

On K8s, `RINGFENCE_CADDY_HOST` is set to the **stable Caddy ClusterIP** (from `helm get values`) rather than a hostname — this eliminates the DNS dependency during rule installation and is safe because ClusterIPs are stable for the lifetime of the Service.

## Environment variables

| Variable | Default | Description |
|---|---|---|
| `RINGFENCE_CADDY_HOST` | `caddy` | Caddy hostname or IPv4 address. K8s: use ClusterIP. |
| `RINGFENCE_CADDY_PORT` | `443` | TCP port for the Caddy ACCEPT rule. |
| `RINGFENCE_DNS_SERVER` | `127.0.0.11` | DNS server ACCEPT (Docker embedded DNS / kube-dns ClusterIP). |
| `RINGFENCE_RUNTIME` | `unknown` | Runtime hint injected by codegen from `YSG_RUNTIME`. |
| `RINGFENCE_AGENT_NAME` | `agent` | Agent name for log prefix (informational). |

## Rootless Podman gap (documented, not papered over)

iptables manipulation inside a rootless Podman user-namespace requires `SYS_ADMIN` or full rootful context, neither of which is granted. When `RINGFENCE_RUNTIME=podman-rootless`:

1. Script logs the gap clearly.
2. Writes `/run/ringfence/l1-gap` marker.
3. Exits **0** (does not block the agent from starting).

The compose codegen annotates the generated stanza:

```yaml
# ROOTLESS-PODMAN-L1-GAP: ringfence-init iptables will not apply.
# L1 network-plane containment is NOT active for this deployment.
# L2 (Caddy egress enforcement via Caddyfile route + OPA) and
# L3 (OPA per-call policy) remain active.
# Upgrade to rootful Podman or K8s for full L1 containment.
```

Upgrade path: `YSG_RUNTIME=podman` (rootful, `sudo podman` context) or K8s.

## IPv6 posture (CRIT-03)

IPv6 is disabled at the compose `ringfence_<agent>` network level:

```yaml
ringfence_<agent>:
  internal: true
  enable_ipv6: false
  driver_opts:
    com.docker.network.bridge.enable_ip_masquerade: "false"
  driver: bridge
  labels:
    yashigani.io/ringfence: "true"
    yashigani.io/agent: "<agent>"
```

`ip6tables OUTPUT DROP` is applied as defence-in-depth for dual-stack K8s environments where the CNI may enable IPv6 despite the compose sysctl.

## Reviewer touchpoints

- **Laura (NetworkPolicy + container-escape):** container-escape post-escape blast radius: if an agent escapes its container, it shares the ringfenced network namespace but iptables in that namespace still blocks direct outbound. Escalation path is host network (requires `--network host` which Kyverno blocks) or CAP_NET_ADMIN (dropped). Residual risk is escape → ringfence-init namespace → manipulate iptables rules (requires NET_ADMIN at host level, not container level). Mitigated by C7 (Caddy TLS key isolation) + L9 (hardened security context).
- **Nico (cert-mount):** SPIFFE cert is NOT mounted into the init sidecar — only into the agent container. The init sidecar has no secret access. Cert mounts are in the agent compose stanza, not the init stanza.
- **Iris (drift gate):** every PR touching `ringfence-init.sh` or `Dockerfile` triggers a cross-runtime drift audit per `feedback_helm_compose_parity_gate_per_pr`.
